# forensic-audit — Руководство пользователя

Статический аудитор качества кода, предоставленный в виде MCP-сервера. Вы указываете ему проект, он читает код через `tree-sitter` / AST и сообщает о проблемах. Он **никогда не запускает ваш код** — никаких импортов, никакого выполнения тестов, никаких побочных эффектов.

## Что это такое

`forensic-audit` — это аудитор исходного кода, работающий только на чтение. Он разбирает файлы в синтаксические деревья и обходит их в поисках известных проблемных шаблонов. Поскольку это чисто статический анализ, его безопасно запускать на коде, которому вы не доверяете, и на коде, который вы не можете (или не хотите) выполнять.

Поддерживаемые языки:

- **Python** — полная поддержка (самые глубокие проверки нацелены на Python).
- **Go, Java, JavaScript, TypeScript** — частичная поддержка через адаптеры (мельче, чем для Python).

## Что вы получаете

Он обнаруживает **22 класса проблем** (полнота 22/22 на тестовом корпусе проекта):

- SQL-инъекция (заражённая строка передана напрямую в `.execute()`)
- Захардкоженные секреты
- `eval` / `exec`
- `shell=True` / `os.system`
- Небезопасные `pickle` / `yaml.load`
- Изменяемые аргументы по умолчанию
- Утечки ресурсов (`open` без блока `with`)
- Широкий / голый `except`, который проглатывает ошибки
- Мёртвый код (приватный, на который нет ссылок)
- Неиспользуемые импорты
- Магические числа
- `TODO` / `FIXME`
- Отладочный `print`
- Закомментированный код
- «Наивный» `datetime` (без таймзоны)
- Конкатенация путей
- Пропущенный `await`
- Непроверенный ответ HTTP
- Слишком большие функции
- Глубокая вложенность
- Почти дублирующийся код

Вывод устроен **по принципу «сначала сводка»**: по умолчанию вы получаете компактную сводку (~3k токенов) — количество по уровням серьёзности, количество по `check_id` и топ находок из наивысшего присутствующего уровня. Запрашивайте `view='full'` только тогда, когда вам нужен полный, постраничный список.

На уровне запуска возвращается `exit_code`:

- `0` — чисто
- `1` — есть находки уровня HIGH / CRITICAL
- `2` — ошибка

Вывод **детерминирован**: один и тот же код даёт одни и те же находки от запуска к запуску.

## Когда использовать

- Перед коммитом или слиянием изменения.
- Когда нужно второе мнение о качестве кодовой базы и её зоне риска.
- При аудите стороннего или незнакомого кода, который вы предпочли бы не запускать.
- Как проход, ориентированный на полноту (recall): он склонен сообщать о большем, поэтому хорош для сценария «покажи мне всё, что может быть не так», после чего вы проводите триаж.

Это **не** средство запуска тестов и **не** замена линтера для чисто стилистических вопросов — он сосредоточен на реальных багах, проблемах безопасности и структурных «запахах».

## Как установить и использовать

На данный момент это **локальная установка в editable-режиме** (на PyPI его пока нет). Он работает на той машине, где вы его установили; публикация на PyPI для использования на других машинах ожидается.

Из корня репозитория `cortex-codeintel`:

```bash
pip install -e .   # из репозитория cortex-codeintel
claude mcp add forensic-audit -s user -- <abs-path-to-venv-python> -m cortex_mcp.forensic_server
```

Замените `<abs-path-to-venv-python>` на абсолютный путь к интерпретатору Python в вашем виртуальном окружении, например в Windows:

```
C:\Users\You\path\to\cortex-codeintel\.venv\Scripts\python.exe
```

После добавления сервер предоставляет API по схеме **фоновая задача + опрос**. Типичный поток таков:

1. `start_forensic_audit(path)` → возвращает `job_id` и `resolved_path`. Оставьте `path` пустым, чтобы автоматически определить корень проекта, поднимаясь вверх от текущего каталога.
2. `get_forensic_status(job_id)` → опрашивайте, пока `status == "done"` (обычно секунды).
3. `get_forensic_results(job_id)` → компактная сводка. Читайте её первой.
4. Только при необходимости: `get_forensic_results(job_id, view='full', severity='HIGH')` или `check_id='...'`, чтобы углубиться в конкретные находки.

Чего ожидать на реальном коде:

- На чистой небольшой библиотеке (например, `filelock`, ~14 файлов) он сообщает примерно **38 находок**, в основном настоящих — большой размер файла, широкий `except`, неиспользуемые импорты.
- На крупных проектах он выдаёт множество находок (он ориентирован на полноту). **Сначала разбирайте HIGH.**

Корень проекта определяется автоматически, если вы опускаете `path`.

## Плюсы

- **Полная полнота по 22 классам** на тестовом корпусе.
- **Низкий уровень ложных срабатываний на чистом коде.**
- **Настраиваемость** — можно включать, отключать или ограничивать набор запускаемых проверок (см. «Тонкая настройка»).
- **Компактная сводка**, которая не раздует ваш бюджет контекста.
- **Кросс-платформенность.**
- **Детерминированный** вывод, стабильный от запуска к запуску.

## Минусы и ограничения

Честно о том, чего он **не** делает:

- **Обнаружение SQL-инъекций узкое.** Оно срабатывает только тогда, когда заражённая строка является **прямым** аргументом `.execute()`. Отслеживания «заражения» (taint-tracking) через переменные нет, поэтому инъекция, собранная через промежуточные переменные, может быть пропущена.
- **Сканирование может занимать ~20с на очень больших проектах** (обход AST по каждой проверке).
- **Уровень ложных срабатываний на больших кодовых базах не проверялся построчно.** Инструмент ориентирован на полноту, так что на крупных проектах ждите шума и проводите триаж соответственно.
- **Глубина неравномерна по языкам.** Некоторые проверки глубже всего для Python; Go / Java / JS / TS — мельче.
- Он **не** запускает ваш код, поэтому всё, что проявляется только во время выполнения, вне его охвата. (Для структурного понимания — точки входа, кто что пишет, карты зависимостей — используйте сопутствующий сервер `code-map`.)

## Тонкая настройка

Конфигурация хранится в `<project>/.cortex/`.

- **Отключить шумные проверки** для проекта: создайте `<project>/.cortex/disabled_gates.json` со списком идентификаторов проверок, например `["gate_id", ...]`. Эти проверки никогда не запускаются.
- **Запустить только определённые проверки:** передайте `gates="id1,id2"` (через запятую). Всё остальное пропускается.
- **Поднять порог серьёзности:** передайте `severity="HIGH"`, чтобы оставить только находки HIGH / CRITICAL.
- **Включить опциональные эвристические проверки** (выключены по умолчанию из-за шума, например `god_object_zones`): укажите их в `gates=`.
