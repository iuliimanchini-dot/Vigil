# forensic-audit — 用户指南

一个以 MCP 服务器形式提供的静态代码质量审计工具。你把它指向一个项目，它就用 `tree-sitter` / AST 读取代码并报告问题。它**从不运行你的代码**——不导入、不执行测试、没有任何副作用。

## 它是什么

`forensic-audit` 是一个只读的源代码取证审计工具。它将文件解析成语法树并遍历这些树，查找已知的问题模式。由于它是纯静态分析，因此在你不信任的代码上、以及在你无法（或不愿）执行的代码上运行都是安全的。

支持的语言：

- **Python**——完整支持（最深入的检查项针对 Python）。
- **Go、Java、JavaScript、TypeScript**——通过适配器提供部分支持（比 Python 浅）。

## 它能给你什么

它可检测 **22 类问题**（在项目测试语料上召回率为 22/22）：

- SQL 注入（被污染的字符串直接作为 `.execute()` 的参数传入）
- 硬编码的密钥
- `eval` / `exec`
- `shell=True` / `os.system`
- 不安全的 `pickle` / `yaml.load`
- 可变的默认参数
- 资源泄漏（`open` 时没有使用 `with` 块）
- 宽泛 / 裸 `except`，会吞掉错误
- 死代码（私有且无任何引用）
- 未使用的 import
- 魔法数字（magic numbers）
- `TODO` / `FIXME`
- 调试用的 `print`
- 被注释掉的代码
- 不带时区的 `datetime`（naive datetime）
- 路径拼接
- 缺失的 `await`
- 未检查的 HTTP 响应
- 过大的函数
- 过深的嵌套
- 近似重复的代码

输出**以摘要为先**：默认你得到一份精简摘要（约 3k tokens）——按严重级别统计、按 `check_id` 统计，以及当前出现的最高严重级别中的头部问题。只有当你需要完整、分页的列表时，才请求 `view='full'`。

每次运行在整体层面会带有一个 `exit_code`：

- `0`——干净
- `1`——存在 HIGH / CRITICAL 级别的问题
- `2`——错误

输出是**确定性的**：相同的代码在多次运行中产生相同的结果。

## 何时使用

- 在提交或合并改动之前。
- 当你想对某个代码库的质量和风险面获得第二意见时。
- 在审计你宁可不执行的第三方或不熟悉的代码时。
- 作为一次以召回为导向的全面扫描：它倾向于多报，因此适合“把所有可能有问题的地方都给我列出来”的场景，然后由你来分诊。

它**不是**测试运行器，也**不是**仅处理风格问题的 linter 替代品——它聚焦于真实的 bug、安全问题和结构性的代码异味。

## 如何安装与使用

目前这是一个**本地可编辑安装（editable install）**（尚未发布到 PyPI）。它在你安装它的那台机器上可用；发布到 PyPI 以便在其他机器上使用的工作仍在进行中。

在 `vigil` 仓库根目录下：

```bash
pip install -e .   # 在 vigil 仓库中执行
claude mcp add forensic-audit -s user -- <abs-path-to-venv-python> -m vigil_mcp.forensic_server
```

请将 `<abs-path-to-venv-python>` 替换为你虚拟环境中 Python 解释器的绝对路径，例如在 Windows 上：

```
C:\Users\You\path\to\vigil\.venv\Scripts\python.exe
```

添加之后，该服务器提供一套**后台任务 + 轮询**的 API。典型流程如下：

1. `start_forensic_audit(path)` → 返回一个 `job_id` 和 `resolved_path`。将 `path` 留空可从当前目录向上查找以自动检测项目根目录。
2. `get_forensic_status(job_id)` → 轮询直到 `status == "done"`（通常只需几秒）。
3. `get_forensic_results(job_id)` → 精简摘要。请先读这一份。
4. 仅在需要时：`get_forensic_results(job_id, view='full', severity='HIGH')` 或 `check_id='...'`，以深入查看特定问题。

在真实代码上可以预期：

- 在一个干净的小型库上（例如 `filelock`，约 14 个文件），它会报告大约 **38 个问题**，大多是真实的——文件过大、宽泛 `except`、未使用的 import。
- 在大型项目上它会产生很多问题（它以召回为导向）。**请先分诊 HIGH。**

省略 `path` 时，项目根目录会被自动检测。

## 优点

- 在测试语料上具有**完整的 22 类召回**。
- 在干净代码上**误报率低**。
- **可配置**——可以启用、禁用或限制运行哪些检查项（见“调优”）。
- **精简摘要**，不会撑爆你的上下文预算。
- **跨平台**。
- **确定性**输出，多次运行保持稳定。

## 缺点与局限

诚实地说明它**做不到**什么：

- **SQL 注入检测范围很窄。** 仅当被污染的字符串是 `.execute()` 的**直接**参数时才会触发。它不跨变量做污点追踪（taint-tracking），因此通过中间变量拼接而成的注入可能被漏掉。
- **在非常大的项目上，一次扫描可能耗时约 20 秒**（逐检查项的 AST 遍历）。
- **在大型代码库上的误报率尚未逐行核实。** 它以召回为导向，因此在大项目上要预期会有噪声，并据此分诊。
- **各语言的深度并不均衡。** 部分检查项对 Python 最深入；Go / Java / JS / TS 则较浅。
- 它**不**运行你的代码，因此任何仅在运行时才暴露的问题都不在其覆盖范围内。（若需要结构性理解——入口点、谁写了什么、依赖关系图——请使用配套的 `code-map` 服务器。）

## 调优

配置位于 `<project>/.cortex/` 之下。

- **为某个项目禁用嘈杂的检查项：** 创建 `<project>/.cortex/disabled_gates.json`，内容为一组检查项 id 的列表，例如 `["gate_id", ...]`。这些检查项将永不运行。
- **只运行特定的检查项：** 传入 `gates="id1,id2"`（以逗号分隔）。其余的都会被跳过。
- **提高严重级别下限：** 传入 `severity="HIGH"`，只保留 HIGH / CRITICAL 级别的问题。
- **启用可选的启发式检查项**（默认关闭，因为较吵，例如 `god_object_zones`）：在 `gates=` 中写出它们的名字。
