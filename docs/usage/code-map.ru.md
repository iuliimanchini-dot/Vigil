# code-map — Руководство пользователя

Построитель структурных карт кодовых баз, предоставленный в виде MCP-сервера. Вы указываете ему проект, он читает код через `tree-sitter` / AST и строит карты того, как код организован. Как и сопутствующий аудитор, он **никогда не запускает ваш код**.

## Что это такое

`code-map` — это построитель статических карт, работающий только на чтение. Он разбирает исходники в синтаксические деревья и выдаёт структурированные карты: что что импортирует, какие символы определены, где находятся точки входа во время выполнения, какие есть контракты данных и как они «дрейфуют», кто пишет в файлы / БД / окружение, где зоны риска и где пролегают естественные границы рефакторинга. Он предназначен для *понимания* кодовой базы, а не для поиска багов (для этого используйте сервер `forensic-audit`).

Поддерживаемые языки: **Python, Go, Java, JavaScript, TypeScript**.

## Что вы получаете

Он строит **8 типов карт**, из которых **6 работают «из коробки»** без какой-либо настройки:

- **structural** — импорты и символы. Точная карта.
- **data_contract** — dataclass-ы / структуры, их форма, обнаружение дрейфа, а также те, кто пишет и читает их (writers / readers).
- **hotspot** — оценка риска для каждого файла.
- **conflict** — конфликты схем.
- **authority** — кто пишет в файлы / БД / окружение. Автоматически выявляет записи без всякой настройки.
- **runtime** — точки входа (`__main__` / async). Автоматически выявляются без настройки.

Седьмая карта, **refactor_boundary**, **управляется seed-ом**: для работы ей нужна цель рефакторинга. (Восьмая, `findings`, является производной от остальных.)

Вывод устроен **по принципу «сначала сводка»**: по умолчанию вы получаете количество по каждому типу карты плюс топ-записи каждой карты. Запросите одну полную карту через `map='structural'` или `view='full'` для всех карт сразу.

Вывод **детерминирован** — семантический diff игнорирует временны́е метки, так что один и тот же код даёт одни и те же карты. Результаты **кэшируются на диске** в `<project>/.cortex/`, благодаря чему повторные запуски дёшевы.

## Когда использовать

- Когда вы попадаете в незнакомую кодовую базу и нужно быстро понять её архитектуру.
- Чтобы узнать, что что импортирует или от чего зависит.
- Чтобы найти точки входа во время выполнения или зоны риска.
- Чтобы оценить объём рефакторинга (а при наличии seed-а — найти его границы).
- Чтобы увидеть, кто пишет в файлы / базы данных / окружение — карта authority выявляет места записи автоматически.

Это **не** аудитор качества кода — для багов, проглоченных исключений и проблем безопасности используйте сервер `forensic-audit`.

## Как установить и использовать

На данный момент это **локальная установка в editable-режиме** (на PyPI его пока нет). Он работает на той машине, где вы его установили; публикация на PyPI для использования на других машинах ожидается.

Из корня репозитория `cortex-codeintel`:

```bash
pip install -e .   # из репозитория cortex-codeintel
claude mcp add code-map -s user -- <abs-path-to-venv-python> -m cortex_mcp.map_server
```

Замените `<abs-path-to-venv-python>` на абсолютный путь к интерпретатору Python в вашем виртуальном окружении, например в Windows:

```
C:\Users\You\path\to\cortex-codeintel\.venv\Scripts\python.exe
```

После добавления сервер предоставляет API по схеме **фоновая задача + опрос**. Типичный поток таков:

1. `start_code_map(path)` → возвращает `job_id` и `resolved_path`. Оставьте `path` пустым, чтобы автоматически определить корень проекта, поднимаясь вверх от текущего каталога.
2. `get_code_map_status(job_id)` → опрашивайте, пока `status == "done"`.
3. `get_code_map_results(job_id)` → компактная сводка (количество по картам + топ-записи). Читайте её первой.
4. Углубление: `get_code_map_results(job_id, map='structural')` для одной полной карты или `view='full'` для всех карт (обе постранично через `page=`).

Чтобы перечитать карты, построенные в предыдущей сессии, без запуска новой задачи, используйте `load_code_map_by_path(path)`.

Корень проекта определяется автоматически, если вы опускаете `path`.

## Плюсы

- **6 из 7 карт работают без какой-либо настройки.**
- **Богатая карта `data_contract`** с обнаружением дрейфа.
- **Точная карта structural.**
- **5 языков** (Python, Go, Java, JS, TS).
- **Детерминированность** — семантический diff игнорирует временны́е метки.
- **Сводка укладывается в ваш бюджет контекста.**

## Минусы и ограничения

Честно о пробелах:

- **`refactor_boundary` требует seed** (цель рефакторинга). Без него он не выдаёт границ.
- **Обнаружение для карты `authority` в Python неполное.** Она ловит `open('w')`, `write_text` и `json.dump`, но **не** ловит идиому атомарной записи `os.fdopen` + `os.replace`, а также записи в БД / окружение. (Адаптеры Go / Java записи в БД ловят.)
- **Некоторые точки входа не выявляются.** Точки входа, объявленные только через `console_scripts` в упаковке (без `__main__` в самом файле), не обнаруживаются.
- **Полный вид одной карты может быть большим** — до ~32k символов на проекте среднего размера. Он постраничный, так что листайте его, а не ждите всё сразу.

## Тонкая настройка

- Большинству карт настройка **не нужна** — 6 из 7 работают «из коробки».
- Единственное исключение — **`refactor_boundary`**: задайте цель рефакторинга в качестве seed-а, чтобы он знал, какую границу вычислять.
- Используйте **сначала сводку**, затем углубляйтесь в одну карту через `map='<type>'` (например, `map='structural'`) или получайте всё через `view='full'`. Оба вида постраничные через `page=`.
- Результаты **кэшируются на диске** в `<project>/.cortex/`; `load_code_map_by_path(path)` перечитывает их без новой сборки.
